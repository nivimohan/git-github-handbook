Introduction

Imagine you are writing a 100-page novel. Every day you make changes — some brilliant, some terrible. Now imagine accidentally deleting three chapters with no way to recover them. That nightmare is what developers faced before version control systems existed.

Git solves this problem for code. It tracks every change you make to your files, lets you travel back in time to any previous state, and allows multiple people to work on the same codebase simultaneously — without overwriting each other's work.

GitHub is the online platform built on top of Git. It stores your code in the cloud, makes collaboration effortless, and serves as the world's largest hub for open-source software.

Pro Tip

Git is the tool; GitHub is the platform. You can use Git without GitHub, but GitHub without Git is impossible. Master the tool first.

What is Version Control?

Version Control Systems (VCS) are software tools that track and manage changes to source code over time. They act like a time machine — you can see who changed what, when, and why. You can also revert files to a previous state at any time.

🏢 Centralized VCS (CVCS)
  • Single central server stores all history
  • Developers must connect to server to work
  • Examples: SVN, CVS, Perforce
  • Risk: If server goes down, everyone is blocked
  • Risk: Single point of failure for all data
🌐 Distributed VCS (DVCS)
  • Every developer has a full copy of the history
  • Work offline — sync when ready
  • Examples: Git, Mercurial
  • Advantage: No single point of failure
  • Advantage: Faster operations (all local)

Chapter 02

Git vs GitHub

Understanding the difference — a confusion that trips up almost every beginner.

The single most common misconception for beginners is thinking Git and GitHub are the same thing. They are not. They are complementary tools that work together.

🔧 Git
  • A command-line tool installed on your computer
  • Tracks changes to files over time
  • Works completely offline
  • Created by Linus Torvalds (2005)
  • Open source & free forever
  • Runs on Windows, macOS, Linux
🐙 GitHub
  • A website that hosts Git repositories
  • Adds collaboration features (PRs, Issues)
  • Requires internet connection
  • Founded by Tom Preston-Werner (2008)
  • Owned by Microsoft (since 2018)
  • Free tier + paid plans available
Good to Know

GitHub is not the only option. Alternatives include GitLab, Bitbucket, and Gitea. They all work with Git — the tool is always Git, the platform varies.

Fork vs Clone

🍴 Fork
  • Copies someone else's repo to your GitHub account
  • Happens entirely on GitHub (online)
  • Used to contribute to open-source projects
  • Your fork is completely independent
  • You can submit a Pull Request back to the original
📥 Clone
  • Copies a GitHub repository to your local computer
  • Happens via the git clone command
  • Used to start working on a project locally
  • Maintains a link to the original (remote)
  • You can push changes back if you have permission

SSH vs HTTPS

Protocol URL Format Authentication Best For
HTTPS https://github.com/user/repo.git Username + Personal Access Token Quick setup, firewalls, beginners
SSH git@github.com:user/repo.git SSH key pair (no password needed) Daily work, more secure, professionals
Best Practice

Set up SSH keys for your daily workflow. Once configured, you never need to enter a password again. Use HTTPS for quick, one-off operations on unfamiliar machines.

Chapter 03

Git Configuration

Setting up your identity and preferences — the first thing to do after installing Git.

Before you can use Git, you must tell it who you are. Git uses this information to sign every commit you make. Think of it as filling in your name before submitting a document.

⚙️ git config

Set Your Identity (Required)

bash 
$ git config --global user.name "Your Name"
$ git config --global user.email "you@example.com"

Set Default Editor & Branch Name

bash 
$ git config --global core.editor "code --wait"  # VS Code
$ git config --global init.defaultBranch main

View Your Configuration

bash 
$ git config --list
user.name=Your Name
user.email=you@example.com
core.editor=code --wait
init.defaultBranch=main
Scope Levels

--system = all users on PC
--global = your user account
--local = this repo only

Config File Locations

Global: ~/.gitconfig
Local: .git/config
System: /etc/gitconfig

.gitignore — Exclude Files from Git

The .gitignore file tells Git which files and folders to completely ignore. This is essential for keeping your repository clean and secure.

.gitignore 
# Dependencies
node_modules/
vendor/

# Build output
dist/
build/
*.class

# Secrets — NEVER commit these!
.env
*.pem
secrets.json

# OS files
.DS_Store
Thumbs.db

README.md — Your Project's Front Page

A README.md file is the first thing people see when they visit your GitHub repository. It is written in Markdown and should explain what your project is, how to install it, and how to use it. A great README dramatically increases your project's credibility.

Common Mistake

Never commit sensitive data like API keys, passwords, or tokens. Always add .env files to .gitignore before your very first commit. Once committed, secrets are extremely difficult to fully remove from Git history.

Chapter 04

Git Clone

Download a complete copy of any repository to your local machine.

Cloning creates a full local copy of a remote repository — including all files, branches, commits, and history. It also automatically sets up the remote connection so you can push and pull changes.

📥 git clone
bash 
# Clone into a folder named after the repository
$ git clone https://github.com/username/repository.git

# Clone into a specific folder name
$ git clone https://github.com/username/repository.git my-project

# Clone using SSH (recommended for daily use)
$ git clone git@github.com:username/repository.git

# Clone only a specific branch
$ git clone --branch develop https://github.com/username/repo.git

Example Output

output 
Cloning into 'repository'...
remote: Enumerating objects: 125, done.
remote: Counting objects: 100% (125/125), done.
remote: Compressing objects: 100% (78/78), done.
Receiving objects: 100% (125/125), 48.23 KiB | 2.10 MiB/s, done.
Resolving deltas: 100% (42/42), done.
What it does

Downloads the full repo, all branches, all commits, and sets origin remote automatically.

When to use it

When starting work on an existing project for the very first time.

Pro Tip

After cloning, run cd repository-name to enter the project folder, then git log --oneline to quickly view the commit history.

Chapter 05

Git Status

Always know where you stand — the most frequently used Git command.

git status shows you a snapshot of your working directory. It tells you which files are modified, which are staged, and which are untracked. You should run this command constantly as you work.

🔍 git status
bash 
$ git status

On branch main
Your branch is up to date with 'origin/main'.

Changes to be committed:
  (use "git restore --staged <file>..." to unstage)
        modified:   index.html

Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
        modified:   style.css

Untracked files:
  (use "git add <file>..." to include in what will be committed)
        script.js

Short Format

bash 
$ git status -s
M  index.html  # Staged modified
 M style.css   # Unstaged modified
?? script.js   # Untracked

Reading the Status Output

Status Meaning Action Needed
Untracked New file Git has never seen Run git add <file>
Modified (unstaged) Changed but not yet staged Run git add <file>
Staged Ready for the next commit Run git commit
Nothing to commit Working directory is clean No action needed ✓
Pro Tip

Make git status a reflex. Run it before git add, before git commit, and before git push. It takes 1 second and can save you hours of confusion.

Chapter 06

Git Add

Move changes from your working directory into the staging area.

Before Git can save your changes, you must stage them using git add. Think of it like putting items into a shopping cart before checking out. You choose exactly which changes go into each commit.

➕ git add
bash 
# Stage a single file
$ git add index.html

# Stage multiple specific files
$ git add index.html style.css app.js

# Stage all changes in the current folder
$ git add .

# Stage all tracked modified files
$ git add -u

# Interactively choose what to stage (patch mode)
$ git add -p
Common Mistake

Using git add . without checking git status first. You might accidentally stage build files, logs, or secrets. Always review what you are adding.

Pro Tip

Use git add -p (patch mode) to stage individual hunks within a file. This lets you make one commit for a bug fix and a separate commit for a feature — even if both changes are in the same file.

Chapter 07

Git Commit

Save a permanent snapshot of your staged changes to the repository history.

A commit is like a save point in a video game. It records the exact state of your project at a specific moment in time. Every commit has a unique ID (SHA hash), the author's identity, a timestamp, and a message describing what changed.

💾 git commit
bash 
# Standard commit with a message
$ git commit -m "feat: add navigation bar to homepage"

# Stage all tracked changes AND commit in one step
$ git commit -am "fix: typo in README"

# Amend the last commit (fix message or add forgotten file)
$ git commit --amend -m "feat: corrected commit message"

Example Output

output 
[main 4a9c2f1] feat: add navigation bar to homepage
 2 files changed, 47 insertions(+), 3 deletions(-)

Writing Great Commit Messages

Type When to Use Example
feat: A new feature feat: add dark mode toggle
fix: A bug fix fix: resolve login redirect loop
docs: Documentation only docs: update README setup guide
style: Formatting, no logic change style: format with prettier
refactor: Code restructure, no behavior change refactor: extract auth helper
test: Adding or fixing tests test: add unit tests for cart
chore: Build tools, dependencies chore: upgrade webpack to v5

Viewing Commit History

bash 
$ git log --oneline
4a9c2f1 (HEAD -> main) feat: add navigation bar
7f3b2e8 fix: typo in README
1d4a0c7 docs: initial commit

# Full graph with all branches
$ git log --oneline --graph --all --decorate
Common Mistake

Never use git commit --amend on a commit that has already been pushed to a shared branch. It rewrites history and will cause problems for anyone else who has pulled that commit.

Chapter 08

Git Push

Upload your local commits to the remote repository on GitHub.

After committing changes locally, you use git push to upload them to the remote repository so others can see and access your work. Think of it as pressing "Send" after writing an email.

⬆️ git push
bash 
# Push the current branch to origin
$ git push

# Push and set upstream (first push of a new branch)
$ git push -u origin main

# Push a specific branch
$ git push origin feature/login

# Push all branches
$ git push --all origin

# Force push with safety check
$ git push --force-with-lease
When push fails

If someone else pushed before you, your push will be rejected. Run git pull first to merge their changes, then push again.

Force push (Dangerous!)

git push --force overwrites remote history. Never use on shared branches. Prefer --force-with-lease.

Warning — Force Push

git push --force on a shared branch will erase other people's work. Only use it on your own private branches. Always prefer git push --force-with-lease which checks no one else pushed first.

Chapter 09

Git Pull & Git Fetch

Keep your local repository in sync with the remote.

git pull — Download and Merge

git pull does two things in one step: downloads the latest changes from the remote (git fetch) and merges them into your current branch (git merge).

⬇️ git pull
bash 
# Pull and merge from the tracked remote branch
$ git pull

# Pull from a specific remote and branch
$ git pull origin main

# Pull using rebase instead of merge (cleaner history)
$ git pull --rebase origin main

git fetch — Download Without Merging

git fetch downloads new changes from the remote but does not automatically merge them into your local branch. This lets you inspect incoming changes before applying them.

🔄 git fetch
bash 
# Fetch all changes from origin
$ git fetch origin

# Preview what was fetched before merging
$ git log HEAD..origin/main --oneline

# Manually merge the fetched changes
$ git merge origin/main
⬇️ git pull
  • One command = fetch + merge
  • Faster for routine sync
  • Can cause unexpected merge conflicts
  • Best when you trust the remote changes
🔄 git fetch
  • Download only — you decide when to merge
  • Safer — inspect before integrating
  • Good for reviewing team changes first
  • Best in fast-moving team environments
Best Practice

On shared team branches, use git fetch + git log HEAD..origin/main to preview incoming changes before merging. This prevents surprise merge conflicts.

Chapter 11

Branching

Work on features and fixes in complete isolation without affecting the main codebase.

Branching is one of Git's most powerful features. A branch is an independent copy of your project where you can make changes safely — without touching the main working codebase. When you are done, you merge it back.

Real-world analogy: Imagine a science lab. You have a master formula. Instead of experimenting on the master, you make photocopies (branches). You run experiments on copies. If an experiment works, you update the master.

Git Branching Visualization

C1 ─── C2 ─── C3 ─── C4 ────────── C7 ← main (merged)
C1 ─── C2 └─── C5 ─── C6 ← feature/login
🌿 git branch & git switch
bash 
# List all local branches (* = current)
$ git branch
* main
  develop

# List all branches including remote
$ git branch -a

# Create AND switch to a new branch (modern syntax)
$ git switch -c feature/login

# Switch to an existing branch
$ git switch main

# Old syntax (still works)
$ git checkout -b feature/signup

# Delete a fully merged branch
$ git branch -d feature/login

# Force delete an unmerged branch
$ git branch -D feature/experiment

Branch Naming Conventions

Pattern Purpose Example
feature/<name> New features feature/user-auth
fix/<name> Bug fixes fix/login-redirect
hotfix/<name> Urgent production fixes hotfix/payment-crash
docs/<name> Documentation changes docs/api-reference
release/<version> Release preparation release/v2.1.0
Best Practice

Keep branches small and focused on a single task. A branch for "redesigning the homepage and fixing the auth bug and updating dependencies" is too broad. Create three separate branches instead.

Chapter 12

Merging

Combine the work from one branch into another.

Merging is the process of taking the changes from one branch and integrating them into another. This is how features get incorporated into the main codebase.

🔀 git merge
bash 
# Step 1: Switch to the branch you want to merge INTO
$ git switch main

# Step 2: Merge the feature branch
$ git merge feature/login
Updating 7f3b2e8..4a9c2f1
Fast-forward
 src/login.js | 52 +++++++++++++++++++++++++++
 1 file changed, 52 insertions(+)

# Merge with an explicit merge commit (no fast-forward)
$ git merge --no-ff feature/login -m "Merge login feature"

# Abort a merge if conflicts arise
$ git merge --abort
⚡ Fast-Forward Merge
  • Happens when no new commits on base branch
  • Git simply moves the pointer forward
  • No merge commit created
  • Cleaner linear history
🔀 Three-Way Merge
  • Both branches have new commits
  • Git creates a special merge commit
  • Preserves branch history visually
  • Use --no-ff to force this always
Pro Tip — Rebase vs Merge

git rebase rewrites commit history to create a clean, linear log. Use it on local feature branches before merging, but never rebase a branch that others are using.

Chapter 13

Pull Requests

The professional way to propose, review, and merge code changes on GitHub.

A Pull Request (PR) is a GitHub feature — not a Git command — that lets you notify teammates that your feature branch is ready to be reviewed and merged. It is the heart of collaborative development.

Pull Request Workflow

1️⃣ Create a feature branch
↓git switch -c feature/my-feature
2️⃣ Make changes & commit
↓git add . && git commit -m "..."
3️⃣ Push branch to GitHub
↓git push -u origin feature/my-feature
4️⃣ Open Pull Request on GitHub
↓
5️⃣ Code Review by teammates
↓
6️⃣ Approved → Merge & Delete Branch

Writing a Great Pull Request

Section What to Include
Title Clear, concise description using Conventional Commits format
Description Why the change is needed, the approach taken, and any trade-offs
Screenshots Before/after images for any UI changes
Testing How you tested the change and what test cases were added
Linked Issues Use "Closes #123" to auto-close an issue when the PR merges
Remember

A Pull Request means: "I am requesting that you pull my changes into your branch." Despite the name, you are the one pushing — you are asking a maintainer to pull and merge your branch.

Chapter 14

Merge Conflicts

Understand, resolve, and prevent the most feared situation in Git.

A merge conflict happens when two branches have made different changes to the same part of the same file and Git cannot automatically decide which version to keep. This is completely normal — Git simply asks you to make the decision.

What a Conflict Looks Like

conflict markers in index.html 
<h1>
<<<<<<< HEAD
  Welcome to My App
=======
  Hello World — My Awesome App
>>>>>>> feature/new-homepage
</h1>

Conflict Resolution Steps

⚠️ Conflict detected during git merge
↓git status
View all conflicting files
↓
Open conflicting file(s) in editor
↓
Remove conflict markers. Keep the correct code.
↓git add <resolved-file>
Stage the resolved file
↓git commit -m "Resolve merge conflict"
✓ Conflict resolved — merge complete
bash — Resolving a Conflict 
# After manually editing the conflicted file:
$ git add index.html
$ git commit -m "fix: resolve merge conflict in index.html"

# Or if you want to abort and start over:
$ git merge --abort
Pro Tip — Prevent Conflicts

Pull from the main branch frequently while working on your feature branch. The longer you wait, the worse conflicts get. Small, frequent pulls are the single best way to avoid large conflicts.

↩️ git restore — Undo Changes Safely
bash 
# Discard unstaged changes to a file (PERMANENT)
$ git restore index.html

# Unstage a file (move back from staging to working dir)
$ git restore --staged index.html

git reset — Move HEAD and Undo Commits

Command Effect on Commits Effect on Files Use Case
git reset --soft HEAD~1 Undo last commit Files stay staged Fix commit message or add forgotten file
git reset HEAD~1 Undo last commit Files unstaged but kept Redo commit with different staging
git reset --hard HEAD~1 Undo last commit Files DELETED Completely discard a bad commit
Warning — Hard Reset

git reset --hard permanently deletes uncommitted changes. There is no undo. Never use it on shared branches.

⏪️ git revert — Safely Undo a Pushed Commit

Unlike git reset, git revert creates a new commit that undoes a previous one. This is safe to use on public branches because it does not rewrite history.

bash 
# Revert the last commit
$ git revert HEAD

# Revert a specific commit by SHA hash
$ git revert 4a9c2f1
📦 git stash — Temporarily Save Work

Stash saves uncommitted changes to a temporary stack and restores a clean working directory. Perfect for quickly switching branches without committing unfinished work.

bash 
# Save current work-in-progress
$ git stash

# Save with a description
$ git stash push -m "half-finished login form"

# List all stashes
$ git stash list
stash@{0}: WIP on main: half-finished login form

# Apply AND remove from stash list
$ git stash pop

How professional teams structure their Git usage for efficient collaboration.

Complete Git Development Workflow

📋 Pick a task (from Issues / Sprint board)
↓
🌿 Create a feature branch from main
↓git switch -c feature/task-name
✏️ Write code. Make small, frequent commits.
↓git add . && git commit -m "feat: ..."
⬆️ Push your branch to GitHub
↓git push -u origin feature/task-name
🔍 Open a Pull Request on GitHub
↓
👥 Code Review — make requested changes
↓
✓ PR approved → Merge to main
↓git branch -d feature/task-name
🚀 Deploy (triggered by CI/CD)

Git Flow vs GitHub Flow

🔄 Git Flow
  • Uses: main, develop, feature/*, release/*, hotfix/*
  • Structured release cycle
  • Good for versioned software (mobile apps, libraries)
  • More complex — more branches to manage
⚡ GitHub Flow
  • Uses: main + short-lived feature/* branches
  • Deploy from main continuously
  • Good for web apps with constant deployment
  • Simple — most teams start here
Best Practice

Start with GitHub Flow. It is simpler, widely adopted, and works for 90% of teams. Only move to Git Flow when you have explicit versioned release requirements.

⚠️ Common Mistakes to Avoid

Learn from the mistakes every beginner makes.

  • 🔐
    Committing secrets and API keys. Never commit .env files, API keys, passwords, or tokens. Use .gitignore from day one. Once pushed, secrets are very hard to fully erase from history.
  • 💬
    Writing vague commit messages. "fix stuff" or "update" tell you nothing later. Write meaningful messages: "fix: resolve null pointer in cart checkout".
  • 🚫
    Working directly on main. Always create a feature branch. Direct commits to main bypass code review and risk breaking the production codebase.
  • 💥
    Using git push --force on shared branches. This overwrites other people's work. Use --force-with-lease or avoid force pushing entirely on shared branches.
  • 📦
    Making huge, monolithic commits. Each commit should do one thing. Giant commits are impossible to review, debug, or revert cleanly.
  • ⏳
    Not pulling before starting new work. Always run git pull before creating a new branch or starting work. Otherwise your branch will be behind and conflicts multiply.
  • 📁
    Committing node_modules. This bloats your repo by megabytes or gigabytes. Always include node_modules/ in .gitignore.
  • ❌
    Using git reset --hard on pushed commits. This rewrites history that others already have. Use git revert instead — it creates a new undo commit that is safe for shared branches.

🚀 Best Practices

Habits that separate junior developers from professionals.

  • Always write a clear, descriptive commit message. Future-you will be grateful.
  • Keep branches short-lived and focused. One branch = one feature or fix.
  • Run git status constantly. Make it a reflex before every command.
  • Always add a .gitignore before your first commit.
  • Use SSH keys instead of typing passwords for remote operations.
  • Write a thorough README.md — it is your project's front door.
  • Review your own PR before asking others — read every line of the diff yourself first.
  • Pull from main frequently to minimize merge conflicts.
  • Tag your releases: git tag v1.0.0 && git push --tags
  • Use GitHub Issues to track all work — nothing should live only in someone's head.
  • Set up a CI/CD pipeline (GitHub Actions) to run tests automatically on every PR.
  • Use branch protection rules on main to require PR reviews before merging.

🎯 Interview Questions & Answers

18 frequently asked Git & GitHub questions for technical interviews.

  • What is Git and why do we use it?
    Git is a distributed version control system that tracks changes to files over time. We use it to maintain a complete history of a project, collaborate with teammates without overwriting each other's work, and safely experiment with new features using branches.
  • What is the difference between Git and GitHub?
    Git is the tool — a command-line program installed on your machine that tracks changes. GitHub is a cloud platform that hosts Git repositories and adds collaboration features like Pull Requests, Issues, and GitHub Actions.
  • Explain the difference between git pull and git fetch.
    git fetch downloads new changes from the remote but does not apply them to your current branch. git pull does both — it fetches and then immediately merges. Use git fetch when you want to inspect changes before merging.
  • What is a merge conflict and how do you resolve it?
    A merge conflict occurs when two branches modify the same lines of the same file differently. To resolve it: open the conflicting file, look for the <<<<<<< HEAD markers, manually edit the file to the correct version, then stage and commit the resolution.
  • What is the difference between git merge and git rebase?
    git merge combines branches by creating a new merge commit, preserving full history. git rebase rewrites feature branch commits as if they were made on top of the target branch, creating a cleaner linear history. Merge is safer for shared branches; rebase is better for cleaning up local history.
  • What is HEAD in Git?
    HEAD is a pointer that indicates the commit you are currently working on — essentially "where you are" in your repository. Normally it points to the latest commit of your current branch. A "detached HEAD" state means HEAD points directly to a commit rather than a branch.
  • What is the difference between git reset and git revert?
    git reset moves the HEAD pointer backward, rewriting history (dangerous on shared branches). git revert creates a new commit that undoes a previous one without changing history — it is safe to use on public branches.
  • What is the staging area (index)?
    The staging area is an intermediate area between the working directory and the repository. When you run git add, files move to the staging area. When you run git commit, only what is staged gets committed. This gives you precise control over what goes into each commit.
  • How do you undo the last commit without losing your changes?
    Use git reset --soft HEAD~1. This moves HEAD back one commit but keeps all your changes staged, allowing you to re-commit with a different message or add more files.
  • What is a Pull Request?
    A Pull Request (PR) is a GitHub feature where you notify teammates that your feature branch is ready for review and merging. It provides a space for code review, comments, discussion, and automated CI checks before the code is merged into the main branch.
  • What is the difference between fork and clone?
    A fork copies a repository to your GitHub account (online operation on GitHub). A clone copies a repository to your local computer (via git clone). You typically fork a project to contribute to it, then clone your fork to work on it locally.
  • What does git stash do?
    git stash temporarily saves uncommitted changes to a stack and restores the working directory to a clean state. You can later retrieve your work with git stash pop. It is useful when you need to switch branches quickly without committing unfinished work.
  • How do you delete a remote branch?
    Run git push origin --delete branch-name. After merging a Pull Request on GitHub, there is also a "Delete branch" button that does this automatically. Remember to also delete the local copy with git branch -d branch-name.
  • What is .gitignore?
    A .gitignore file specifies intentionally untracked files that Git should not monitor or commit. Common entries include node_modules/, .env, build outputs, and OS-specific files like .DS_Store.
  • What is a fast-forward merge?
    A fast-forward merge happens when the branch being merged into has no new commits since the feature branch was created. Instead of creating a merge commit, Git simply moves the branch pointer forward, creating a clean linear history.
  • How do you see the commit history?
    Use git log for the full history. Use git log --oneline for a compact view. Use git log --oneline --graph --all --decorate for a visual branch graph in the terminal.
  • What is the difference between git diff and git diff --staged?
    git diff shows changes in the working directory that have NOT been staged yet. git diff --staged shows changes that ARE staged — i.e., what will be included in the next commit.
  • Explain the concept of "origin" in Git.
    origin is the conventional default name given to the remote repository when you clone or add a remote. It is just an alias for the full URL. You can have multiple remotes with different names. upstream is the conventional name for the original repository when you have forked someone else's project.

📌 Summary & Key Takeaways

Everything you should remember after reading this handbook.

  • Git is a distributed version control tool. GitHub is a cloud platform for hosting and collaborating on Git repositories.
  • The three core areas are: Working Directory → Staging Area → Repository.
  • The daily workflow is: git status → git add → git commit → git push.
  • Always work on a feature branch — never commit directly to main.
  • Write clear, atomic commit messages. One commit = one logical change.
  • Use git fetch to preview changes and git pull to download and merge.
  • Merge conflicts are normal. Resolve them by editing the file, staging, and committing.
  • Use git revert on public branches and git reset only on local commits.
  • Always add a .gitignore before your first commit. Never commit secrets or tokens.
  • Pull Requests are the professional standard for code review and collaboration.
  • GitHub Actions, Issues, Projects, and Pages extend Git into a full DevOps platform.
